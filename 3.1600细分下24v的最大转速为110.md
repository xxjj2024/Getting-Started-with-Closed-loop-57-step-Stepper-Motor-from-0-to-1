## 疑问
为什么此处不用pid算法来让低速平滑过渡到高速呢。
## pid和滤波算法的比较

我们可以用“老司机开车”的比喻来彻底拆解它们：

- **滤波算法（Filtering）：** 是老司机的**“眼睛和大脑”**。马路上的测速仪可能坏了，忽快忽慢（传感器噪声），滤波算法负责把这些虚假、抖动的数据剔除，还原出真实的当前车速。
    
- **PID 算法（Proportional-Integral-Derivative）：** 是老司机的**“手和脚”**。它根据滤波后得到的真实车速，决定当下是该踩油门、踩刹车，还是保持方向盘不动。
    

在实际应用中，它们通常结合在以下几个最核心的场景中：

---

### 场景一：拯救疯狂的“D”项（微分滤波）

这是 PID 和滤波结合**最普遍、最致命**的用武之地。

在标准的 PID 公式中，**D（微分项）** 的作用是“预测未来”。它通过计算误差的变化率（当前误差减去上一次误差），来提前踩刹车，防止超调。

**现实的灾难：**

现实世界中的传感器（比如电机的编码器、温度计）永远是有“毛刺”噪声的。哪怕真实速度很平稳，传感器读数也可能在 0.001 秒内从 60 瞬间跳到 61，再跳回 60。

- 对于 P（比例）和 I（积分）来说，这点微小的跳动无所谓。
    
- **但对于 D 来说，这是毁灭性的！** D 项对高频噪声极其敏感。微小的数值跳动经过微分计算后，会被无限放大。结果就是，单片机会疯狂地输出极端信号，导致电机剧烈抖动、发出尖锐的啸叫，甚至烧毁驱动器。
    

**组合用法：**

在工业级的 PID 算法中，我们绝对不会直接使用原始的 D 项。我们会在 D 项的计算路径上，强行加入一个**一阶低通滤波器（Low-Pass Filter）**。

$$D_{filtered}(k) = \alpha \cdot D_{raw}(k) + (1-\alpha) \cdot D_{filtered}(k-1)$$

_($\alpha$ 是滤波系数，决定了我们是更相信当前的新数据，还是更相信历史的老数据。)_

加了滤波后的 D 项，就像戴上了墨镜，无视了那些刺眼的高频闪烁，电机瞬间变得无比平顺。

---

### 场景二：传感器信号的“前置净化”（输入端滤波）

有时候，不仅是 D 项受不了噪声，连最基础的 P 项也被噪声干扰得无法工作。这时候我们需要在数据进入 PID 算法**之前**，先做一次大清洗。

**典型应用领域：**

1. **无人机姿态控制：** 无人机上的陀螺仪和加速度计，无时无刻不在感受电机高速旋转带来的剧烈高频震动。如果直接把这种抖成筛子的角度数据喂给 PID，无人机直接就在空中翻滚炸机了。必须先经过**卡尔曼滤波（Kalman Filter）**或互补滤波，算出极其平滑的真实姿态，再交给 PID 去控制四个螺旋桨。
    
2. **温控系统（烤箱/3D打印机喷头）：** ADC（模数转换器）读取热敏电阻时，极容易受到周围电磁环境的干扰，读数会在 199°C 和 201°C 之间疯狂横跳。我们通常会在 PID 之前加一个**滑动窗口平均滤波（Moving Average Filter）**，取最近 10 次的平均值，得到稳定的 200°C 后，再去算 PID。
    
3. **电机测速反馈：** 读取编码器脉冲算速度时，如果计算周期极短，读到的速度值会呈现锯齿状波动。必须经过滤波平滑后，PID 才能稳定地把速度稳在目标值。
    

---

### 场景三：防止“设定值突变”的冲击（设定值滤波 / 柔性给定）

这是极其高级的玩法。

假设你的电机正在以 60转/分 稳定运行（PID 处于平衡状态）。突然，上位机下发了一个指令，要求瞬间变成 200转/分。

- **如果不加滤波：** PID 算法一算，误差瞬间变得极其巨大！P 项会立刻输出 100% 的最大功率，导致机械结构受到极其暴力的冲击（打齿、断轴）。
    
- **加入设定值滤波：** 我们不让 PID 直接看到“200”这个突变的数字。我们用一个滤波算法（或者斜坡函数），让目标值从 60 缓慢、平滑地爬升到 200。PID 追着这个平滑移动的目标跑，整个加速过程就会像高铁起步一样丝滑。
**理论上极其正确，但在现在的这套硬件架构下，千万不要这么做！**

这里面藏着一个工业控制里极其经典的“架构级认知误区”。我们把这层窗户纸捅破，你的控制理论水平会瞬间跃升一个台阶。

---

### 误区揭秘：你这套系统的“真面目”

在工业界，电机控制分为两大派系。我们来看看你属于哪一派：

#### 派系一：单片机当“苦力”（原始 PID 控制）

- **硬件配置：** STM32 + 纯傻瓜驱动板（比如 L298N） + 普通直流电机 + 编码器（直接连到 STM32 的引脚上）。
    
- **控制逻辑：** STM32 必须自己读取编码器算出现实速度，然后自己跑 PID 算法，算出来一个 PWM 占空比，直接控制电机的电压。
    
- **在这个派系里，你的想法完全正确！** 从 45转 到 80转，STM32 会用斜坡函数生成目标速度，然后用 PID 紧紧咬住这个目标。
    

#### 派系二：单片机当“甩手掌柜”（也就是你现在的架构）

- **硬件配置：** STM32 + **UMLP57 闭环驱动器** + 闭环步进电机（编码器连在驱动器上，不连 STM32）。
    
- **控制逻辑：** 你的 UMLP57 驱动器内部有一个极其强大的 DSP 芯片。**那个芯片里面，已经在以每秒几万次的频率狂跑极其精密的工业级 PID 算法了！**
    

### 为什么你不能在 STM32 里再写一个 PID？

**1. 你没有“眼睛”（缺少反馈）** PID 的核心是“误差 = 目标 - 现实”。但是，你电机的编码器是插在 UMLP57 驱动器上的，你的 STM32 根本不知道电机现在的真实速度是多少。没有现实数据，PID 就成了瞎子，根本算不出任何东西。

**2. “两个老司机抢方向盘”（级联震荡）** 退一万步讲，就算你把编码器信号分了一根线给 STM32，让 STM32 也跑 PID 去动态修改 `ARR`。 这就变成了：你的 STM32 在跑一套 PID，底层的 UMLP57 也在跑一套 PID。一旦外界有干扰，STM32 决定加速，UMLP57 决定减速，两个环路就会互相打架。在工业上，这叫**“级联控制回路耦合震荡”**，你的电机会像抽风一样疯狂哆嗦，然后彻底死锁报警。

---

### 恍然大悟：其实“早就”写了斜坡函数！

你回过头来看看你之前写的这几行代码：

```
// 缓慢加速到 80转 (ARR从 1200 降到 74)
for(int arr = 1200; arr >= 74; arr -= 3) 
{
    Motor_SetSpeed_ARR(arr); 
    Delay_ms(3);
}
```

**你以为你只是写了一个普通的 `for` 循环？** **不！在工业机器人的底层代码里，这就叫做“梯形速度轨迹规划（Trapezoidal Velocity Profile）”，也就是我们刚才说的“设定值滤波 / 斜坡函数”！**

在这套架构里，职责划分得极其明确：

- **你的 STM32 是“总参谋部”（轨迹规划器）：** 它的任务不是去管底层的电流和扭矩，而是用 `for` 循环（斜坡函数）平滑地改变发送脉冲的频率（也就是不断发布新的平滑目标速度）。
    
- **UMLP57 是“一线作战部队”（底层 PID 执行者）：** 它接收到你 STM32 发来的脉冲频率（平滑移动的目标），然后利用它自己内部的闭环 PID 算法，控制电流，死死地咬住你给的这个频率，保证电机不丢步。
