### 核心一：为什么 TTL 通信必须用“中断 (Interrupt)”？

其实 TTL 通信，在 STM32 底层叫 **USART（通用同步/异步收发器）**，俗称串口通信。

**1. 如果不用中断（灾难现场）：** 想象一下，你的 `main` 函数死循环（老板）正在全神贯注地盯着电机，执行那段加减速的代码（里面有大量的 `Delay`）。 此时，你的上位机突然通过 TTL 线发来一个极其紧急的指令：“STOP（立刻停止）！” 但是，老板正在忙着执行 `Delay`，根本没空去门外的信箱（USART接收寄存器）看信。等老板忙完了再去拿信，指令早就被后面的数据覆盖了，或者机器早就撞坏了。这就叫**“轮询（死等）模式的通信丢失”**。

**2. 用了中断（完美配合）：** 在 STM32 里，我们要开启一个叫 **`USART_IT_RXNE`（接收数据寄存器非空中断）** 的东西。

- **物理机制：** 这个中断就像是给秘书配了一个“高音喇叭”。老板（CPU）依然在专心致志地控制电机加减速。
    
- **动作发生：** 只要上位机通过 TTL 线发来**哪怕一个字节**的数据，秘书立刻按下高音喇叭（触发中断）。
    
- **瞬间响应：** CPU 无论在干什么（哪怕在 `Delay` 里），都会**瞬间被迫停下手中的活**，跑到特定的“中断服务函数”里，把这封信（指令）拿出来存进数组里。存好之后，CPU 瞬间闪现回去，接着控制电机。
    
- **结果：** 电机转得极其丝滑，且绝不会漏掉上位机发来的任何一条指令！这叫真正的**异步并行处理**。
    

---

### 核心二：看门狗 (Watchdog) 在这里扮演什么救命角色？

一旦引入了上位机和通信线缆，系统的脆弱性就会成倍增加。

**1. 通信线缆脱落的灾难：** 假设你的上位机发了一条指令：“电机全速正转！” STM32 收到了，电机开始狂奔。就在这时，意外发生了——连接上位机和下位机的 **TTL 杜邦线被扯断了**，或者上位机的软件突然卡死了！ 你的 STM32 现在成了一个瞎子和聋子，它永远收不到“停止”指令了，传送带将永远以全速狂奔，直到把机械撞毁。

**2. 引入“通信看门狗（心跳包机制）”：** 真正的工业控制，都会在这个时候用上**看门狗（IWDG 或独立定时器）**配合“心跳包”。

- **规则：** 规定上位机每隔 1 秒，必须通过 TTL 线给下位机发一个“我还活着 (Heartbeat)”的闲聊指令。
    
- **喂狗：** STM32 的串口中断每次收到这个指令，就去“踢”一下看门狗（把看门狗定时器清零）。
    
- **救命瞬间：** 如果 TTL 线断了，或者上位机死机了。STM32 超过 2 秒没收到心跳包，没去喂狗。看门狗倒计时结束，“咔嚓”一刀切断底层逻辑，强制 STM32 瞬间切断电机脉冲，并亮起故障红灯。
    
- **这就叫“断线保护”！**
## 同步中断和异步中断比较
### 一、 同步 (Synchronous) —— “死等”与“排队”

**核心逻辑：** 一件事情没做完，CPU 绝对不去做下一件事。代码是严格按顺序、一行一行往下死磕的。这在编程里叫**“阻塞 (Blocking)”**。

**现实比喻（同步点餐）：**

你去前台点了一份炸鸡（调用接收数据的函数）。

前台说：“正在炸，请稍等。”

于是，**你就死死地站在柜台前面盯着油锅（死循环等待）**。在这 10 分钟里，你不能刷手机，不能去上厕所，不能跟朋友聊天。直到炸鸡交到你手上，你才转身去做下一件事。

**STM32 代码里的灾难现场（轮询模式）：**

如果你用“同步”的方式去接收上位机发来的串口指令，代码长这样：
```
// 同步接收：CPU 在这里死等，直到收到上位机的一个字母
while (USART_GetFlagStatus(USART1, USART_FLAG_RXNE) == RESET) 
{ 
    // 里面什么都不写，CPU 就在这原地踏步发呆
}
char cmd = USART_ReceiveData(USART1); // 终于等到了，拿走数据

Motor_SetSpeed_ARR(80); // 然后才去控制电机
```

**致命后果：** 如果上位机 1 个小时都不发指令，你的 CPU 就会在这个 `while` 循环里卡死 1 个小时！外面的电机要是正在狂奔，早撞烂了，因为 CPU 根本没空去执行刹车代码。

---

### 二、 异步 (Asynchronous) —— “统筹”与“见机行事”

**核心逻辑：** 发起一个任务后，CPU 不等结果，直接转头去干别的重要事情。等那个任务有结果了，通过一个**“信号（中断）”**通知 CPU 过来处理。这叫**“非阻塞 (Non-blocking)”**。

**现实比喻（异步点餐）：**

你去前台点了一份炸鸡。

前台给你一个**震动取餐器**（配置硬件中断）。

然后你立刻转身找个座位坐下，**开始和朋友打游戏、聊天（CPU 执行主循环里的电机加减速控制）**。

过了 10 分钟，炸鸡好了，取餐器“嗡嗡”一震（触发串口接收中断）。你**瞬间按了游戏的暂停键**，花 10 秒钟跑去柜台把炸鸡拿过来放在桌上，然后**马上恢复游戏继续打**。

**STM32 代码里的完美配合（中断模式）：**

在单片机里，这是一种**“硬件 + 软件”的真实并行处理**。

1. **独立部门：** STM32 内部的 USART 串口硬件，是一个独立的部门。无论 CPU 在干嘛，串口硬件都在默默监听 TTL 线上的电平变化。
    
2. **老板（CPU）干活：** CPU 在 `while(1)` 里专心致志地算加减速、发脉冲。
    
3. **高音喇叭（中断）：** 上位机发来一个字母，串口硬件瞬间收到，立刻按响警报（触发 `USART1_IRQHandler` 中断函数）。
    
4. **瞬间切断：** CPU 放下电机的活，以微秒级的速度跳进中断函数，把字母存进内存（拿炸鸡）。
    
5. **光速返回：** 存完立刻跳回刚才电机算到一半的代码，继续执行。因为速度太快了，电机连抽搐一下的时间都没有。
    

---

### 终极对比总结

|**特性**|**同步 (Synchronous)**|**异步 (Asynchronous / 中断)**|
|---|---|---|
|**代码逻辑**|事情必须一件一件按顺序做|事情可以交叉着做|
|**CPU状态**|经常在“死等”（极度浪费算力）|永远在干最有价值的活|
|**实时响应**|极差（如果卡在 Delay 里，根本不响应）|**极高（发生突发事件，微秒级响应）**|
|**工业适用性**|只能点亮个 LED 灯玩玩|**商业级机器人的绝对基石**|
## 上位机控制步骤
### 第一步：修路（物理层的硬件接线）

通信的第一步是得有物理连接。电脑只有 USB 口，单片机只有 TTL 串口，语言不通怎么办？

我们需要一个**“翻译官”**，也就是 **USB 转 TTL 模块**。

- **动作：** 把模块插在电脑 USB 口上。
    
- **接线原则（交叉接法）：**
    
    - 电脑要说话（模块的 `TXD`） $\rightarrow$ 连单片机的耳朵（STM32 的 `PA10 / RX`）。
        
    - 电脑要听话（模块的 `RXD`） $\rightarrow$ 连单片机的嘴巴（STM32 的 `PA9 / TX`）。
        
    - 统一电平参考：模块的 `GND` 必须连单片机的 `GND`。
        
- **完成状态：** 物理公路修好了，电平信号可以互相跑了。
    

---

### 第二步：开设通信部门（底层初始化配置）

单片机刚上电的时候，PA9 和 PA10 只是普通的引脚，它们听不懂串口信号。你必须在 `main` 函数的最开始（初始化阶段），填几张表，把它们升级为“专业通信部门”。

我们需要配置三个东西：

1. **GPIO 引脚配置：**
    
    - 告诉 `PA9`：“你现在不是普通工人了，你的身份是‘复用推挽输出’，专门负责往外发信号。”
        
    - 告诉 `PA10`：“你的身份是‘浮空输入’，专门负责在门口监听别人发来的电平。”
        
2. **USART 串口配置：**
    
    - 设定双方聊天的“语速”（波特率）。比如设为 `115200`。这就好比规定大家讲话都是每分钟 100 个字，谁也别嫌谁快，不然就会听到一堆乱码。
        
3. **NVIC 中断控制器配置（最核心的报警器）：**
    
    - 告诉单片机内部的安保大队长：“给串口 1 发一个特权。只要 PA10 收到电脑发来的哪怕一个字母，你要立刻拉响顶级警报，把 CPU 叫过来！”
        

- **完成状态：** 部门成立了，并且配备了收到消息就响的“高音喇叭（中断）”。
    

---

### 第三步：制定“收信规则”（编写中断服务函数）

这是完全独立于 `main` 函数之外的一个特殊函数。它的名字是被官方写死的，叫 `USART1_IRQHandler`。

- **它的角色：** 它是老板（CPU）的贴身秘书。
    
- **它的特点：** 平时它处于“休眠”状态。只有当电脑发来数据，触发了警报（中断），CPU 才会被迫瞬间跳进这个函数里执行。
    
- **它的工作逻辑（极其简单）：**
    
    1. 确认是不是真的收到数据了？
        
    2. 从硬件信箱里把那个字母拿出来。
        
    3. **把它放在桌面上（存进一个全局变量里）。**
        
    4. 转身离开，让 CPU 赶紧回去控制电机。
        
- **完成状态：** 单片机具备了在做任何事情时，瞬间接住电脑指令的能力，并且把指令存在了内存里。
    

---

### 第四步：老板的执行逻辑（改造 `while(1)` 主循环）

现在，我们回到 `main` 函数那个永远在跑的死循环里。

之前，我们的主循环是按照死剧本在跑（转3秒，停1秒）。现在我们要把它改造成一个**“听话的执行者”**。

- **角色：** 它是公司的核心业务逻辑（老板）。
    
- **它的工作方式：** 老板每一圈循环，都会低头看一眼秘书放在桌子上的那个**“全局变量”**。
    

**逻辑流：**

- 此时全局变量是 `'0'` $\rightarrow$ 老板判断：没有任何指令，继续跑之前的电机脉冲。
    
- 突然，电脑发来了一个字母 `'A'`（启动指令）。
    
- **（瞬间发生的事情）：** 警报响 $\rightarrow$ 秘书把桌上的变量改成了 `'A'` $\rightarrow$ 警报解除。
    
- 老板再次低头一看，发现全局变量变成 `'A'` 了！
    
- 老板立刻执行：`Motor_SetSpeed_ARR(80); Motor_SetDir(1); Motor_SetState(1);` （调用电机驱动代码，机器瞬间转起来）。
    
- 老板执行完后，把桌上的变量重新改回 `'0'`（表示指令已处理），继续下一圈循环。
    

---

### 梳理总结（为什么要这么设计？）

把这四步串起来，你看懂这套架构的精妙之处了吗？

它把**“接收数据（极快、不可预测）”**和**“控制电机（较慢、需要连续性）”**这两件事，通过一个**“全局变量（桌面）”**完美地隔离开了。

- **中断（秘书）** 只负责眼疾手快地收数据，绝不碰电机。
    
- **主循环（老板）** 只负责稳重地看变量、调电机，绝不死等数据。

### 底层代码

```c
#include "stm32f10x.h"

// ==========================================
// 【核心架构：老板办公桌上的“全局变量”】
// 加 volatile 极其重要！它告诉编译器：这个变量会被中断（秘书）随时修改，
// 老板每次看它必须去内存里重新读，绝对不能凭记忆偷懒。
// ==========================================
volatile uint8_t g_ReceiveCmd = 0; 

// ==========================================
// 提前声明所有的工具和底层 API
// ==========================================
void Delay_ms(uint32_t ms);
void Motor_Init(void);
void Motor_SetState(uint8_t state);
void Motor_SetDir(uint8_t dir);
void Motor_SetSpeed_ARR(uint16_t arr_value);
void UART1_Init_With_Interrupt(void); // 【新部门】串口初始化函数

// ==========================================
// 总裁办：主函数 (只会看桌上的变量来发号施令)
// ==========================================
int main(void)
{
    // 1. 底层部门初始化
    Motor_Init();               // 成立电机控制部
    UART1_Init_With_Interrupt();// 成立通信联络部（配置 PA9/PA10 和中断秘书）

    // 2. 默认开机状态：电机锁死待命
    Motor_SetState(0);

    // 3. 异步死循环：老板永远在这里巡视
    while(1)
    {
        // 如果秘书在桌上放了新指令 (不是 0)
        if (g_ReceiveCmd != 0) 
        {
            // 老板开始解析指令
            switch (g_ReceiveCmd)
            {
                case 'A': // 如果电脑发来大写字母 A (前进指令)
                    Motor_SetDir(1);
                    // 用咱们推导的绝对安全起步法
                    Motor_SetSpeed_ARR(1200); 
                    Motor_SetState(1);
                    for(int arr = 1200; arr >= 99; arr -= 3) {
                        Motor_SetSpeed_ARR(arr); Delay_ms(3);
                    }
                    break;

                case 'B': // 如果电脑发来大写字母 B (后退指令)
                    Motor_SetDir(0);
                    Motor_SetSpeed_ARR(1200); 
                    Motor_SetState(1);
                    for(int arr = 1200; arr >= 99; arr -= 3) {
                        Motor_SetSpeed_ARR(arr); Delay_ms(3);
                    }
                    break;

                case 'S': // 如果电脑发来大写字母 S (刹车指令 Stop)
                    // 同样要敬畏物理，平滑降速后锁死
                    // 注意：这里为了简化，假设当前是 99 的速度。实际工程中可以读取当前 ARR
                    for(int arr = 99; arr <= 1200; arr += 3) {
                        Motor_SetSpeed_ARR(arr); Delay_ms(3);
                    }
                    Motor_SetState(0);
                    break;
            }
            
            // 老板执行完指令后，把桌上的字条扔进垃圾桶（清零）
            // 准备迎接秘书送来的下一个指令
            g_ReceiveCmd = 0; 
        }

        // 如果 g_ReceiveCmd 是 0，CPU 就在这里以极快的速度空转。
        // 此时底层的定时器硬件正在默默地给电机发脉冲，互不干扰！
    }
}

// ==========================================
// 【全新核心】：中断服务函数（秘书专属通道）
// 名字绝对不能写错，这是 STM32 硬件底层写死的！
// ==========================================
void USART1_IRQHandler(void)
{
    // 保安队长确认：真的是因为“接收到了数据(RXNE)”触发的警报吗？
    if(USART_GetITStatus(USART1, USART_IT_RXNE) != RESET)
    {
        // 秘书动作 1：把信箱里的字母拿出来，直接拍在老板桌上（全局变量）
        g_ReceiveCmd = USART_ReceiveData(USART1);
        
        // 秘书动作 2：清除报警灯，告诉硬件“信我拿走了”，准备收下一封
        USART_ClearITPendingBit(USART1, USART_IT_RXNE);
    }
}

// ==========================================
// 【全新核心】：初始化串口和中断部门
// ==========================================
void UART1_Init_With_Interrupt(void)
{
    GPIO_InitTypeDef GPIO_InitStructure;
    USART_InitTypeDef USART_InitStructure;
    NVIC_InitTypeDef NVIC_InitStructure;

    // 1. 给串口 1 (高速网 APB2) 和 引脚 GPIOA 通电
    RCC_APB2PeriphClockCmd(RCC_APB2Periph_USART1 | RCC_APB2Periph_GPIOA, ENABLE);

    // 2. 配置 PA9 (TX 发送嘴巴)：复用推挽输出，外包给硬件
    GPIO_InitStructure.GPIO_Pin = GPIO_Pin_9;
    GPIO_InitStructure.GPIO_Speed = GPIO_Speed_50MHz;
    GPIO_InitStructure.GPIO_Mode = GPIO_Mode_AF_PP;
    GPIO_Init(GPIOA, &GPIO_InitStructure);

    // 3. 配置 PA10 (RX 接收耳朵)：浮空输入，静静监听
    GPIO_InitStructure.GPIO_Pin = GPIO_Pin_10;
    GPIO_InitStructure.GPIO_Mode = GPIO_Mode_IN_FLOATING;
    GPIO_Init(GPIOA, &GPIO_InitStructure);

    // 4. 填串口配置表 (经典 115200 8N1)
    USART_InitStructure.USART_BaudRate = 115200; // 语速
    USART_InitStructure.USART_WordLength = USART_WordLength_8b; // 8位数据
    USART_InitStructure.USART_StopBits = USART_StopBits_1;      // 1位停止位
    USART_InitStructure.USART_Parity = USART_Parity_No;         // 无校验
    USART_InitStructure.USART_HardwareFlowControl = USART_HardwareFlowControl_None;
    USART_InitStructure.USART_Mode = USART_Mode_Rx | USART_Mode_Tx; // 既能听又能说
    USART_Init(USART1, &USART_InitStructure);

    // 5. 开启串口接收中断触发器 (只要收到数据，就给 NVIC 保安队长发信号)
    USART_ITConfig(USART1, USART_IT_RXNE, ENABLE);

    // 6. 填 NVIC 保安队长配置表
    NVIC_PriorityGroupConfig(NVIC_PriorityGroup_2); // 设定优先级分组
    NVIC_InitStructure.NVIC_IRQChannel = USART1_IRQn; // 专属通道：串口1中断
    NVIC_InitStructure.NVIC_IRQChannelPreemptionPriority = 1; // 抢占优先级 (最高0)
    NVIC_InitStructure.NVIC_IRQChannelSubPriority = 1;        // 响应优先级
    NVIC_InitStructure.NVIC_IRQChannelCmd = ENABLE;           // 允许打断 CPU
    NVIC_Init(&NVIC_InitStructure);

    // 7. 串口总电闸合上！
    USART_Cmd(USART1, ENABLE);
}


// ==========================================
// 隐藏的旧模块：电机底层函数和延时 (无需修改)
// ==========================================
void Motor_Init(void) {
    GPIO_InitTypeDef GPIO_InitStructure;
    TIM_TimeBaseInitTypeDef  TIM_TimeBaseStructure;
    TIM_OCInitTypeDef  TIM_OCInitStructure;
    RCC_APB2PeriphClockCmd(RCC_APB2Periph_GPIOA | RCC_APB2Periph_AFIO, ENABLE);
    RCC_APB1PeriphClockCmd(RCC_APB1Periph_TIM2, ENABLE);
    GPIO_InitStructure.GPIO_Pin = GPIO_Pin_0;
    GPIO_InitStructure.GPIO_Mode = GPIO_Mode_AF_OD;
    GPIO_InitStructure.GPIO_Speed = GPIO_Speed_50MHz;
    GPIO_Init(GPIOA, &GPIO_InitStructure);
    GPIO_InitStructure.GPIO_Pin = GPIO_Pin_1;
    GPIO_InitStructure.GPIO_Mode = GPIO_Mode_Out_OD;
    GPIO_Init(GPIOA, &GPIO_InitStructure);
    GPIO_SetBits(GPIOA, GPIO_Pin_1); 
    TIM_TimeBaseStructure.TIM_Prescaler = 30 - 1;     
    TIM_TimeBaseStructure.TIM_Period = 100 - 1; 
    TIM_TimeBaseStructure.TIM_ClockDivision = 0;
    TIM_TimeBaseStructure.TIM_CounterMode = TIM_CounterMode_Up;
    TIM_TimeBaseInit(TIM2, &TIM_TimeBaseStructure);
    TIM_OCInitStructure.TIM_OCMode = TIM_OCMode_PWM1;
    TIM_OCInitStructure.TIM_OutputState = TIM_OutputState_Enable;
    TIM_OCInitStructure.TIM_Pulse = 50;
```

## 难点代码剖析
### 第一部分：为什么引脚要选“推挽”和“浮空”？

还记得我们控制电机发脉冲时，PA0 选的是“复用**开漏** (AF_OD)”吗？因为那边接了 5V，我们只能做个接地的开关。但是串口通信不一样，它是纯粹的 3.3V 逻辑电平交互。

**1. TX 发送嘴巴（PA9）为什么用“复用推挽 (AF_PP)”？**

- **推挽 (Push-Pull)：** 它的内部就像两个大力士，一个专门负责往上推（强行输出 3.3V 高电平），一个专门负责往下拉（强行输出 0V 低电平）。因为 TTL 通信线缆可能会有一两米长，线上有电阻和干扰，必须用“推挽”这种强劲的输出方式，才能保证发出去的方波信号刀砍斧劈般清晰。
    
- **复用 (Alternate Function)：** 老生常谈了，就是老板不管，把 PA9 这个引脚的控制权交给底层的 USART1 硬件去自动操作。
    

**2. RX 接收耳朵（PA10）为什么用“浮空输入 (IN_FLOATING)”？**

- **浮空的意思是“完全不干预”。** 引脚内部不接任何上拉或下拉电阻，处于一种“悬空”的、极其敏感的高阻抗状态。
    
- **为什么必须这样？** PA10 是听电脑讲话的耳朵。如果 PA10 内部自己带了电压，就会和电脑发过来的电压“打架”，导致读到的信号全错。设为“浮空”，就是让 PA10 100% 顺从外面那根杜邦线上的真实电平。
    

---

### 第二部分：硬件流控制是个什么远古技术？

`USART_InitStructure.USART_HardwareFlowControl = USART_HardwareFlowControl_None;`

- **拆字解密：** `Hardware` (硬件) `FlowControl` (流控制) = `None` (无)。
    
- **大白话：** 在几十年前的老式设备里（比如老式打印机），单片机处理速度极其慢。如果电脑发数据太快，单片机处理不过来，就需要两根额外的信号线（叫 CTS 和 RTS）。单片机会在这两根线上大喊：“喂！我处理不过来了，你先别发了！” 这就叫硬件流控制。
    
- **为什么填 None？** 现在的 STM32 跑在 72MHz，处理你电脑打字的几个字母简直游刃有余。而且我们的 USB 转 TTL 模块通常只接 `TX`、`RX`、`GND` 三根线，根本没有那两根流控制线。所以必须设为 `None`（不要流控制，直接敞开发）。
    

---

### 第三部分：保安队长（NVIC）的内部等级制度

这一段是决定中断系统能否完美运行的“法律条文”。

**1. `USART_ITConfig(USART1, USART_IT_RXNE, ENABLE);`**

- 这是在给串口的警报器设定**触发条件**。
    
- `RXNE` = Receive Data Register Not Empty（接收数据寄存器非空）。
    
- **大白话：** 告诉硬件，只有当“信箱里确实收到信了（非空）”的时候，才允许你拉响警报去叫老板。
    

**2. `NVIC_PriorityGroupConfig(NVIC_PriorityGroup_2);`**

- **大白话：** 这是保安队长的“分权制度”。STM32 内部有 16 个优先级级别（用 4 个比特位表示）。`Group_2` 的意思是把这 4 个比特位对半分：**前 2 位给“抢占优先级”，后 2 位给“响应优先级”**。
    

**3. 什么是“抢占优先级 (PreemptionPriority)”？**

- `NVIC_InitStructure.NVIC_IRQChannelPreemptionPriority = 1;`
    
- **大白话（绝对特权）：** 假设老板（CPU）现在正在处理一个抢占优先级为 2 的普通中断任务。突然，串口（抢占优先级为 1，数字越小级别越高）来数据了！串口可以**一脚踹开门，强行打断**当前的普通中断，让老板先来处理串口的数据。这就叫“中断嵌套”。
    

**4. 什么是“响应优先级 (SubPriority)”？**

- `NVIC_InitStructure.NVIC_IRQChannelSubPriority = 1;`
    
- **大白话（排队礼让）：** 假设此时有另外一个中断（抢占优先级也是 1）和我们的串口**在同一微秒**拉响了警报。两人级别一样大，谁先上？这时候就比“响应优先级”。谁的响应优先级数字小，谁就先排在前面。但注意，如果别人已经开始处理了，你只能在门口排队等，**不能踹门打断**。
    

_(注：在我们现在的代码里，只有串口这一个中断，所以你填 0、1、2 都可以，它永远是老大。但养成填表的习惯，是为以后系统里有几十个中断时做准备的。)_

**5. 最后的核按钮**

- `NVIC_Init(&NVIC_InitStructure);`：把排班表交给保安队长执行。
    
- `USART_Cmd(USART1, ENABLE);`：拉下串口 1 的总电闸，通信部正式营业！
